# Security Patterns

## Authentication Architecture

Three coexisting auth mechanisms, two token-based (stateless) + one session-based:

### 1. JWT Token (Internal/UI â†’ Backend)
- Header: `TTT_JWT_TOKEN`
- Algorithm: RS256 (RSA), 1-day expiration, **no refresh mechanism**
- Claims: csId, login, officeId, managerId, employeeName, authorities, readOnly
- Generated by TTT backend, requested via `/jwt` endpoint after CAS login
- Grants: `AUTHENTICATED_USER` + role names

### 2. API Token (External Integration)
- Header: `API_SECRET_TOKEN` (also accepts `token` query param)
- Resolved from DB via `DatabaseApiTokenResolver` â†’ token_permissions table
- Two types: EMPLOYEE (1833 tokens, avg 21 perms) and APPLICATION (23 tokens, avg 4.7 perms)
- Grants: **Only** specific `ApiPermission` enum values â€” **NOT** `AUTHENTICATED_USER`

### 3. CAS (Frontend Web UI)
- Standard Spring Security CAS SSO via `CasAuthenticationFilter`
- Separate `SecurityConfiguration` in frontend module
- Results in JWT token generation for subsequent API calls

## Filter Chain Order
```
Request â†’ JwtTokenAuthFilter â†’ ApiTokenAuthFilter â†’ Authorization
```
First match wins. Stateless session policy (no cookies). CORS enabled for all origins.

## AUTHENTICATED_USER Authority

Defined in `TokenConst` â€” by design separates JWT users (full access) from API tokens (permission-scoped). This intentional separation causes the sick leave CRUD accessibility gap (see [[exploration/api-findings/sick-leave-api-testing]]).

## Authorization Patterns

Controllers use `@PreAuthorize` with three patterns:
1. `hasAuthority('AUTHENTICATED_USER')` â€” JWT only (sick leave, some admin)
2. `hasAuthority('SPECIFIC_PERMISSION')` â€” API token only
3. `hasAnyAuthority('AUTHENTICATED_USER', 'SPECIFIC')` â€” Both (vacation, reports)

Pattern 3 is the correct approach but inconsistently applied. Sick leave uses pattern 1 exclusively for mutations.

## Security Findings

1. **No JWT refresh** â€” 1-day expiry, must re-auth via CAS. Risk: poor UX, forced re-login
2. **Inconsistent permission model** â€” Sick leave vs vacation auth patterns diverge
3. **`token` query param fallback** â€” API tokens accepted in URL query string (exposure in logs/referer)
4. **CORS: all origins** â€” Cross-origin requests unrestricted
5. **Self-approval** â€” No segregation of duties check for vacation approval
6. **Secret key in DB** â€” Token secret stored (presumably encrypted) in `token.secret_key`
